# Morning Brief 前端产品设计说明

## 1. 文档目标

本文档定义 Morning Brief 两个前端产品的功能、信息架构、页面逻辑和状态流转：

1. 晨报阅读端：面向每天阅读资讯的用户。
2. 管理后台：面向内容审核、模型配置和流水线管理。

本文档不规定具体视觉风格。设计方案可以重新定义颜色、字体、间距、组件形态和动效，但不得改变核心业务逻辑、状态含义和安全边界。

---

## 2. 产品目标

Morning Brief 不是泛新闻网站，而是一份面向 AI 产品经理的个人每日情报：

- 看 Builders 正在讨论什么。
- 看开发者社区正在构建什么。
- 识别今天真正值得关注的变化。
- 给出最多两个可以采取的具体行动。

阅读端的目标阅读时间为 5～10 分钟。管理后台的目标是让用户可以理解、干预和控制从信息采集到晨报发布的完整流程。

# 第一部分：晨报阅读端

## 3. 阅读端信息架构

```text
最新晨报 /
├── 今日标题与导语
├── 数据状态提示
├── 今日关键信息
├── Builders 在说什么
├── GitHub 今日趋势
├── 下一步行动
└── 历史晨报入口

历史归档 /archive
└── 按日期倒序排列的晨报列表

晨报详情 /brief/:date
└── 指定日期的完整晨报
```

阅读端不提供信息源配置、模型配置、内容审核和复杂筛选能力。

## 4. 最新晨报首页

路由：

```text
/
```

### 4.1 页面必须展示

- 晨报日期。
- 生成时间。
- 今日标题。
- 今日导语。
- 内容完整性状态。
- 今日关键信息。
- Builders 精选主题。
- GitHub Trending 前五。
- 今日行动建议。
- 历史晨报入口。

### 4.2 页面加载逻辑

```text
读取最新已发布晨报
├── 存在：展示完整晨报
├── 不存在当天晨报，但存在历史晨报：展示空状态和最近一期入口
└── 没有任何晨报：展示首次使用空状态
```

阅读端只能展示已发布内容。草稿和已撤下内容不能通过日期路由访问。

### 4.3 数据异常逻辑

- 部分来源失败时继续展示可用内容。
- 模型失败时允许展示确定性降级摘要。
- 当前展示的不是当天数据时，必须标明最近成功日期。
- 数据状态提示不能覆盖或阻断正文阅读。

## 5. 今日关键信息

每天最多展示三条，是页面最高优先级内容。

该区域是事件级的编辑结论，来源可以同时包括 Follow Builders、GitHub Trending 和其他已启用的可靠来源。系统需要合并重复信息，并回答“今天发生了什么”和“为什么重要”，不承担逐字展示原始内容的职责。

每条包含：

- 标题。
- 100～150 字摘要。
- 为什么重要。
- 来源或相关人物。
- 所属类别。
- 来源人物或来源类型文字标识。

### 5.1 交互逻辑

- 按系统和人工审核后的顺序展示。
- 不足三条时按实际数量展示。
- 同一事件的多条动态合并展示。
- 本区域为纯阅读内容，不设置 Hover 状态、点击操作或外部跳转。
- 支撑总结的来源只以人物或来源类型文字标识展示。
- 本区域不展示或暗示存在一篇与 AI 总结逐字对应的“原文”。
- 阅读端不允许重新排序或修改内容。

## 6. Builders 在说什么

内容来自经过过滤、评分和事件聚合后的 Follow Builders Feed。

该区域是人物观点与原始证据层，只使用 Zara Follow Builders 中的 X、Podcast 和 Blog Feed，不混入 GitHub Trending。它与“今日关键信息”允许出现同一事件：前者给出编辑结论，本区域完整保留入选 Builder 的原始表达。

建议主题类型：

- AI Coding 与 Agent。
- AI 产品与设计。
- 创业和产业观察。
- 其他高价值变化。

每个主题包含：

- 主题标题。
- 一句话主题概括。
- 相关人物。
- 核心观点。
- 为什么值得关注。
- 入选内容的完整原文，不截断、不用 AI 摘要替代。
- 原始动态链接。

### 6.1 原文展示规则

- 完整原文直接呈现在晨报首页和日期详情页中，不需要再次点击才能阅读。
- “查看来源”仅用于跳转到 X、Podcast、Blog 或其他原始网站。
- 原文保持原始语言、段落和换行；翻译或编辑摘要必须与原文分开标识。
- 原文不是中文时，生成阶段调用模型完成忠实的简体中文翻译。
- 中文翻译直接放在对应原文下方，原文与翻译之间使用清晰分割线。
- 翻译不得概括、删减或增加观点；需要保留人名、产品名、链接、语气和段落结构。
- 长原文允许分片翻译后按原顺序合并；翻译失败时继续展示原文并把晨报标记为部分完成。
- 同一事件包含多位 Builder 时，逐位展示各自完整原文。
- 原始链接失效时仍保留已经采集并进入晨报的原文，同时标注来源不可访问。

### 6.2 事件合并逻辑

同一事件被多人讨论时，应组织为一个事件：

```text
事件摘要
├── 官方信息
├── 工程视角
├── 产品视角
└── 各来源完整原文与外部链接
```

页面不应连续展示多张内容高度重复的卡片。

## 7. GitHub 今日趋势

固定展示 GitHub Daily Trending 全球榜前五。

每个项目包含：

- GitHub 官方排名。
- 仓库名称与链接。
- 原始项目描述。
- 主要编程语言。
- 今日新增 Star。
- 累计 Star。
- 中文简介。
- 适合什么用户。
- 为什么值得关注。
- 与用户方向的相关程度。

### 7.1 排名与降级逻辑

- 必须保留 GitHub 官方排名。
- 模型只负责解释，不参与重排。
- README 获取失败时继续展示基础信息。
- 不知道项目爆火原因时不得进行推测。
- 单个项目数据缺失不能阻断其他项目展示。

## 8. 下一步行动

每天最多展示两个具体行动建议，例如：

- 阅读一次重要产品发布。
- 体验一个新的 Agent 工具。
- 收藏一个值得进一步了解的 GitHub 项目。

行动建议必须来源于当天内容。没有足够明确的行动时允许不展示该区域。

## 9. 历史晨报

路由：

```text
/archive
/brief/:date
```

归档列表每项包含：

- 日期。
- 当日标题。
- 一句话导语。
- 内容完整性状态。
- 主要主题。

### 9.1 归档逻辑

- 默认按日期倒序排列。
- 点击进入指定日期的完整晨报。
- 只展示已发布晨报。
- 被撤下的晨报不出现在归档中。
- 第一版不需要全文搜索和复杂标签筛选。

## 10. 阅读端状态

| 状态 | 含义 | 页面行为 |
|---|---|---|
| Complete | 所有主要来源和摘要正常 | 正常展示 |
| Partial | 部分信息源失败 | 展示可用内容和缺失来源提示 |
| Fallback | 模型失败，使用降级摘要 | 展示晨报和降级说明 |
| Empty | 尚未生成晨报 | 展示空状态和历史入口 |
| Stale | 展示的是最近成功数据 | 明确标注数据日期 |

# 第二部分：管理后台

## 11. 管理后台信息架构

建议一级导航：

```text
今日概况
内容审阅
晨报编辑
信息源
Prompt
模型与 API
运行记录
系统设置
```

管理后台是流水线控制台，不是通用 CMS。

### 11.1 默认运行模式

系统默认采用无人值守的全自动模式：

```text
定时采集 → AI 筛选与聚合 → AI 生成晨报 → 自动发布网页 → 自动推送微信
```

- 人工审核是可选能力，不是每日流程的必经节点。
- 低置信度内容进入人工审阅队列，但默认不阻断晨报生成。
- 只有全部模型失败、核心来源全部不可用、投递鉴权失败等严重异常才暂停自动流程。
- 用户可以在系统设置中临时开启人工审核闸门；开启后，发布和推送等待人工确认。

## 12. 今日概况

### 12.1 核心指标

- 采集总数。
- 待审阅数量。
- 已保留数量。
- 已过滤数量。
- 已合并数量。
- 最终入选数量。
- GitHub 项目数量。
- 晨报生成状态。
- 网页发布状态。
- 微信推送状态。

### 12.2 流水线阶段

```text
获取 Zara Feed
获取 GitHub Trending
补充 GitHub README
标准化
去重
相关性过滤
事件聚合
模型摘要
生成晨报
构建网页
微信推送
```

每个阶段展示：

- 当前状态。
- 开始时间。
- 结束时间。
- 耗时。
- 输入与输出数量。
- 失败原因或降级信息。

阶段状态包括：等待、运行、成功、部分成功、失败和取消。

### 12.3 快捷操作

- 运行完整流程。
- 仅重新采集。
- 使用已有快照重新处理。
- 仅重新生成摘要。
- 重新构建网页。
- 测试微信消息。
- 发布晨报。

### 12.4 操作规则

- 同一任务运行中不能重复触发。
- 重新生成不能自动推送微信。
- 重新推送微信必须二次确认。
- 操作后立即展示运行状态和结果。
- 长任务必须显示当前阶段，不能只显示无限加载。

## 13. 内容审阅

### 13.1 筛选条件

- 全部。
- 待审阅。
- 已保留。
- 已过滤。
- 已合并。
- 已发布。
- 来源。
- 人物。
- 类别。
- 日期。
- 相关度区间。

### 13.2 内容列表字段

- 作者。
- 来源。
- 原始内容摘要。
- 发布时间。
- 分类。
- 当前状态。
- 综合分。
- 模型判断原因。
- 所属合并事件。

### 13.3 内容详情

- 完整原文。
- 原始链接。
- 原始数据。
- 相关性评分。
- 重要性评分。
- 新颖性评分。
- 可行动性评分。
- 自动分类。
- 保留或过滤原因。
- 所属事件。
- 相似内容。
- 模型处理记录。

### 13.4 人工操作

- 保留。
- 过滤。
- 修改分类。
- 调整评分。
- 加入或移出今日重点。
- 合并到已有事件。
- 创建新事件。
- 从事件中拆出。
- 撤销修改。

### 13.5 内容状态流转

```text
pending
├── kept
├── dropped
└── merged

kept / merged
└── published
```

人工判断优先于模型结果。人工修改后不自动调用模型，而是标记当前晨报需要重新生成。

## 14. 晨报编辑

晨报采用结构化块编辑，不使用传统自由富文本编辑器。

### 14.1 可编辑区域

- 晨报标题。
- 今日导语。
- 今日关键信息。
- Builders 主题。
- GitHub Trending 解读。
- 今日行动建议。

### 14.2 可执行操作

- 编辑标题、摘要和重要性说明。
- 调整主题顺序。
- 加入或移出今日重点。
- 隐藏或恢复主题。
- 修改行动建议。
- 恢复自动生成版本。

### 14.3 预览模式

- 桌面网页预览。
- 手机网页预览。
- 微信消息预览。

三种预览消费同一份晨报数据，只允许渲染方式不同。

#### 微信消息内容尺度

- 固定展示最多三条“今日关键信息”。
- 每条包含标题和约 80～120 个中文字符的概括。
- 概括需要说明发生了什么、为什么值得关注，让用户不打开网页也能大致理解事件。
- 微信不承载完整来源列表、长背景和延伸分析；这些内容通过末尾的完整晨报链接进入网页查看。
- 不使用只有标题和一句标签式短语的极简目录，也不把完整晨报渲染成图片发送。

### 14.4 保存与发布

- 保存草稿。
- 重新生成。
- 标记审核完成。
- 发布到网页。
- 推送微信。
- 撤下网页。
- 查看已发布版本。

以下动作必须相互独立：

```text
保存草稿 ≠ 发布网页
发布网页 ≠ 推送微信
重新生成 ≠ 自动重新发布
```

## 15. 状态模型

不要使用一个字段同时表达生成、编辑、发布和投递状态。

### 15.1 生成质量

```text
complete / partial / fallback
```

### 15.2 编辑状态

```text
draft / reviewed
```

### 15.3 发布状态

```text
unpublished / published / withdrawn
```

### 15.4 投递状态

```text
not_sent / sending / sent / failed
```

## 16. 信息源管理

### 16.1 Zara Feed

三个上游来源：

- X Feed。
- Podcasts Feed。
- Blogs Feed。

每个来源展示：URL、启用状态、最近抓取时间、最近状态、内容数量和最近错误。

### 16.2 Builder 本地控制

- 是否启用。
- 本地权重。
- 默认类别。
- 每日最大入选数。
- 是否忽略转发。
- 是否仅保留原创内容。

这里不能让用户误以为可以直接修改 Zara 的上游 X 账号列表。

### 16.3 GitHub 配置

- 是否启用。
- 榜单周期。
- 获取数量，默认 5。
- 是否读取 README。
- README 最大读取长度。
- GitHub Token 配置状态。
- 最近抓取状态。

## 17. Prompt 管理

Prompt 类型：

- 内容相关性分类。
- 内容评分。
- 事件聚合。
- Builders 摘要。
- GitHub 项目解读。
- 今日重点。
- 行动建议。
- 最终晨报编辑。

每个 Prompt 展示：

- 当前内容。
- 当前版本。
- 使用任务。
- 最近修改时间。
- 历史版本。

允许编辑、保存新版本、使用固定样例测试、回滚和恢复默认值。

修改 Prompt 后不自动重跑正式晨报。每份晨报必须记录自己使用的 Prompt 版本。

## 18. 模型与 API

### 18.1 Provider 管理

每个 Provider 包含：

- 名称。
- 协议。
- Base URL。
- Secret 环境变量名称。
- 启用状态。
- 连通状态。
- 最近错误。

操作包括新增、编辑、停用、测试连接和删除。

### 18.2 密钥安全

后台只显示：

```text
MORNING_BRIEF_LLM_API_KEY
状态：已配置
```

后台不得：

- 返回 API Key 明文。
- 在浏览器本地存储 API Key。
- 在日志中输出 Authorization Header。
- 提供密钥导出功能。

### 18.3 模型管理

- Provider。
- 模型 ID。
- 显示名称。
- 是否启用。
- 是否支持结构化输出。
- 上下文长度。
- 最大输出长度。

### 18.4 任务路由

每个任务配置：

- 主模型。
- 备用模型。
- 超时时间。
- 最大重试次数。

任务包括过滤、评分、聚类、Builders 摘要、GitHub 摘要、今日重点、行动建议和最终晨报。

调用逻辑：

```text
主模型
└── 失败：备用模型
    └── 失败：确定性降级输出
```

## 19. 运行记录

### 19.1 运行列表

- 运行 ID。
- 触发方式。
- 开始与结束时间。
- 总耗时。
- 当前状态。
- 完成阶段。
- 失败阶段。
- 是否使用备用模型。

触发方式包括手动、定时、重试和内容修改后重生成。

### 19.2 运行详情

- 阶段时间线。
- 输入和输出数量。
- API 请求次数。
- Token 使用量。
- 模型耗时。
- 重试记录。
- 错误摘要。
- 原始技术日志。
- 生成的晨报链接。

默认先展示可读摘要，原始技术日志放在次级区域。

## 20. 系统设置

第一版需要：

- 时区。
- 每日生成时间。
- 每日推送时间。
- 晨报最大内容数。
- 微信推送开关。
- 网页自动发布开关。
- 全局暂停自动任务。
- 数据保留天数。

后续可以增加访问权限、备份、导入导出、费用上限和 Token 上限。

# 第三部分：核心业务流程

## 21. 自动生成流程

```text
定时触发
→ 采集 Zara 和 GitHub
→ 保存原始快照
→ 标准化
→ 去重
→ 过滤
→ 事件聚合
→ 评分排序
→ 模型摘要
→ 生成晨报草稿
→ 等待人工审核或自动发布
```

## 22. 人工审核流程

```text
进入今日概况
→ 检查失败和降级状态
→ 审阅待确认内容
→ 修改保留、过滤和合并结果
→ 重新生成晨报
→ 编辑标题和摘要
→ 预览网页与微信
→ 发布网页
→ 推送微信
```

## 23. 模型失败流程

```text
主模型失败
→ 尝试备用模型
→ 备用模型失败
→ 使用确定性摘要
→ 晨报标记 fallback
→ 允许人工编辑和发布
```

## 24. 来源失败流程

```text
单个来源失败
→ 其他来源继续执行
→ 晨报标记 partial
→ 展示缺失来源
→ 不阻断人工审核与发布
```

## 25. 内容修改后的流程

```text
人工修改内容判断
→ 当前晨报标记为需要更新
→ 用户主动选择重新生成
→ 生成新的草稿版本
→ 原已发布版本保持不变
→ 审核后再决定是否重新发布和推送
```

# 第四部分：通用交互要求

## 26. 导航与可理解性

- 每个页面必须明确说明当前位置。
- 按钮使用具体动作名称，例如“重新生成晨报”，避免只写“执行”。
- 空状态必须告诉用户为什么为空以及下一步可以做什么。
- 状态不能只通过颜色表达。
- 长任务必须提供阶段反馈。

## 27. 响应式范围

阅读端必须完整支持：

- 手机。
- 平板。
- 桌面浏览器。

管理后台必须支持桌面浏览器，并保证手机上可以查看状态和执行紧急操作；复杂内容审核和模型配置可以以桌面为主要场景。

## 28. 可访问性

- 所有操作可通过键盘完成。
- 焦点状态清晰可见。
- 触控目标具有足够尺寸。
- 正文具备适合长时间阅读的行宽和行距。
- 支持减少动态效果、减少透明效果和高对比度偏好。

## 29. 不在第一版实现

- 多用户协作和复杂角色权限。
- 阅读端评论、点赞和社交能力。
- 全文知识库搜索。
- 自由拖拽式页面搭建器。
- 在后台中直接读取或导出 API Key。
- 向 Zara 上游动态添加新的 X 账号。
- 复杂费用结算系统。

## 30. 设计交付要求

设计方案至少应覆盖：

1. 阅读端首页桌面版与手机版。
2. 阅读端历史归档和日期详情。
3. Complete、Partial、Fallback、Empty 和 Stale 状态。
4. 管理后台整体导航框架。
5. 今日概况。
6. 内容审阅列表与详情。
7. 晨报编辑与三种预览。
8. 信息源管理。
9. Prompt 管理。
10. 模型、Provider 和任务路由管理。
11. 运行记录列表与详情。
12. 所有关键确认、失败、空状态和运行中状态。
